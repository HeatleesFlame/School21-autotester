# C/C++ Autotest Runner

Автотестер для проверки учебных C/C++ проектов через `stdin/stdout`.

Пользователь указывает только ссылку на Git-репозиторий студента.

Скрипт сам:

1. клонирует репозиторий студента во временную директорию (по умолчанию в `/tmp`);
2. переключается на ветку `develop`;
3. копирует `.clang-format` из проекта студента (`materials/linters/.clang-format`) в директорию исходников;
4. сканирует директорию с исходниками;
5. выводит список найденных `.c`, `.cpp`, `.cc`, `.cxx` файлов;
6. для каждого найденного файла запускает:
   - `clang-format -n --Werror`;
   - `cppcheck`;
   - строгую компиляцию (`-Wall -Wextra -Werror -pedantic`, для C дополнительно линкуется `-lm`);
   - тесты через `stdin/stdout`;
   - проверку памяти через `valgrind`;
7. печатает итоговый отчёт и финальный вердикт.

---

## Требования

Скрипт рассчитан на запуск под Linux.

Необходимые утилиты:

```bash
git
python3
gcc
g++
clang-format
cppcheck
valgrind
```

Версия `clang-format` должна быть строго `18.1.8`. Проверить:

```bash
clang-format --version
```

Если версия другая, скрипт завершится с ошибкой.

### Важно про snap

Если `clang-format`, `cppcheck` или `valgrind` установлены через **snap**, они не имеют доступа к `/tmp` из-за конфайнмента — проверки будут падать с `No such file or directory` и `could not find or open any of the paths given`.

Решение — клонировать проект в домашнюю директорию через `--workdir-root`:

```bash
python3 autotest.py REPO_URL --workdir-root ~
```

Проверить, откуда установлены инструменты:

```bash
readlink -f $(which clang-format cppcheck valgrind)
```

Если пути ведут в `/snap/...`, используйте `--workdir-root ~`.

## Быстрый старт

Из корня репозитория тестера:

```bash
python3 autotest.py https://github.com/student/project.git
```

По умолчанию: ветка `develop`, директория исходников `src`, тесты из `./tests`, режим сопоставления `stem`.

---

## Опции командной строки

```bash
python3 autotest.py REPO_URL [options]
```

| Опция              | По умолчанию   | Назначение                                                   |
|--------------------|----------------|--------------------------------------------------------------|
| `REPO_URL`         | —              | Ссылка на Git-репозиторий студента (обязательный аргумент).  |
| `--branch`         | `develop`      | Ветка, которую проверяем.                                    |
| `--scan-dir`       | `src`          | Директория внутри проекта студента с исходниками.            |
| `--tests-root`     | `./tests`      | Директория с тестами.                                        |
| `--recursive`      | выкл           | Искать исходники в поддиректориях `scan-dir` рекурсивно.     |
| `--test-map`       | `stem`         | Как сопоставлять исходник и папку тестов (см. ниже).         |
| `--workdir-root`   | `/tmp`         | Корень для временной директории клона. Используйте `~` при snap. |
| `--keep-workdir`   | выкл           | Не удалять временную директорию после проверки.              |
| `--no-checkout`    | выкл           | Не делать `git checkout` на ветку.                           |

## Структура репозитория тестера

```text
tester_repo/
├── autotest.py
├── README.md
└── tests/
    └── ...        # тесты, структура зависит от режима --test-map
```

`.clang-format` в репозитории тестера не нужен — он всегда берётся из проекта студента (`materials/linters/.clang-format`).

## Как добавлять тесты

Тесты — это обычные текстовые файлы, разложенные парами:

```text
01.in    # подаётся программе в stdin
01.out   # ожидаемый stdout
02.in
02.out
```

Правила:

- На каждый `.in` обязателен парный `.out` с тем же именем. Нет пары — файл получает `FAIL`.
- Тесты сортируются по имени файла как строки, поэтому нумеруйте с ведущим нулём: `01`, `02`, ... `10`. Иначе `10.in` встанет раньше `2.in`.
- Если программа ничего не читает, `.in` всё равно нужен — оставьте его пустым.
- `stderr` не проверяется: туда пишет valgrind. Отладочный вывод в stderr на результат тестов не влияет.

### Многострочный ввод и вывод

Файл `.in` передаётся программе в stdin целиком, как есть, включая все переводы строк. Файл `.out` сравнивается со всем stdout программы. Внутренние переводы строк учитываются в сравнении.

Пример: программа читает длину и массив, печатает сумму на одной строке и массив на следующей.

`01.in`:

```text
10
4 3 9 0 1 2 0 2 7 -1
```

`01.out`:

```text
8
4 1 2 2 -1
```

---

## Сравнение вывода

По умолчанию сравнение нестрогое: скрипт делает `.strip()` с обеих сторон, поэтому лишние пробелы и переводы строк по краям не влияют (например, финальный `\n` можно не согласовывать). Внутренние пробелы и переносы строк важны: `25` и `2 5` — разный результат.

Чтобы сравнивать символ в символ, в начале `autotest.py` установите:

```python
STRICT_OUTPUT = True
```

## Режимы сопоставления файлов и тестов (`--test-map`)

Режим определяет, в какой папке искать тесты для конкретного исходника.

### `stem` (по умолчанию)

Имя папки тестов = имя файла без расширения.

```text
src/task1.c    -> tests/task1/
src/main.cpp   -> tests/main/
```

Структура тестов:

```text
tests/
├── task1/
│   ├── 01.in
│   └── 01.out
└── main/
    ├── 01.in
    └── 01.out
```

### `relative`

Полный путь без расширения, слэши заменяются на подчёркивания. Удобно, когда исходники лежат в подпапках и имена могут совпадать.

```text
src/task1.c        -> tests/src_task1/
src/lab1/task2.c   -> tests/src_lab1_task2/
```

### `prefix`

Из имени файла берётся ведущий токен вида буквы+цифры (`D07`, `D08`, `Lab3`), он и становится именем папки. Все файлы с одним префиксом используют общий набор тестов.

```text
src/D07_matrix.c   -> tests/D07/
src/D07_vector.c   -> tests/D07/     # тот же префикс -> те же тесты
src/D08_stack.cpp  -> tests/D08/
```

### `prefix-nested`

Тесты сгруппированы по заданию (префикс), а внутри — отдельная папка на каждый файл. У каждого файла свой набор тестов, группировка по дню сохраняется.

```text
src/D07_matrix.c   -> tests/D07/D07_matrix/
src/D07_vector.c   -> tests/D07/D07_vector/
src/D08_stack.cpp  -> tests/D08/D08_stack/
```

Структура тестов:

```text
tests/
├── D07/
│   ├── D07_matrix/
│   │   ├── 01.in
│   │   └── 01.out
│   └── D07_vector/
│       ├── 01.in
│       └── 01.out
└── D08/
    └── D08_stack/
        ├── 01.in
        └── 01.out
```

Имя вложенной папки должно точно совпадать с именем файла без расширения (`D07_matrix.c` -> папка `D07_matrix`).

Запуск:

```bash
python3 autotest.py REPO_URL --test-map prefix-nested
```

Для режимов `prefix` и `prefix-nested` префикс извлекается регулярным выражением `^([A-Za-z]+\\d+)`. Если имя файла не начинается с такого токена, файл получает `FAIL`. Чтобы поддержать другой формат имён, поправьте `PREFIX_PATTERN` в начале `autotest.py`.

## Проверки для каждого файла

| Проверка   | Инструмент     | Условие успеха                                      |
|------------|----------------|-----------------------------------------------------|
| format     | `clang-format` | `clang-format -n --Werror` вернул код 0.            |
| cppcheck   | `cppcheck`     | Статический анализ без ошибок.                      |
| compile    | `gcc`/`g++`    | Компиляция со строгими флагами прошла.              |
| output     | программа      | stdout совпал с `.out` во всех тестах.              |
| memory     | `valgrind`     | Нет утечек и ошибок памяти во всех тестах.          |

Если компиляция не прошла, тесты и valgrind для этого файла пропускаются, а `output` и `memory` отмечаются как `SKIPPED`.

Таймаут одного запуска программы — 5 секунд. При превышении тест помечается `TIMEOUT`.

Каждый исходник компилируется и запускается отдельно. Это подходит для задач, где каждый `.c`/`.cpp` файл — самостоятельная программа с `main`. Для сборки из нескольких файлов (Makefile/CMake) нужна отдельная доработка.

Для C-программ линкуется `-lm`, чтобы работали функции из `math.h` (`sqrt`, `pow` и т.п.).

---

## Вердикты

Файл получает `OK`, если прошли все пять проверок: format, cppcheck, compile, output, memory. Иначе — `FAIL`.

Итоговый `FINAL VERDICT: OK` — если найден хотя бы один файл и все файлы получили `OK`. Иначе `FINAL VERDICT: FAIL`.

Коды возврата:

- `0` — `FINAL VERDICT: OK`;
- `1` — `FINAL VERDICT: FAIL` или инфраструктурная ошибка (нет утилит, не та версия clang-format, не удалось клонировать и т.п.).

---

## Работа с временной директорией

По умолчанию проект клонируется в `<workdir-root>/c_autotest_xxxxx` и удаляется автоматически после проверки.

С флагом `--keep-workdir` директория сохраняется, и в конце печатается её путь:

```text
Workdir was kept: /tmp/c_autotest_xxxxx
```

Полезно для разбора падений: можно зайти в директорию и запустить проверки вручную.
